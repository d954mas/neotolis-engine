# Phase 28: Demo Integration - Context

**Gathered:** 2026-03-20
**Status:** Ready for planning

<domain>
## Phase Boundary

Sponza demo proving the full v1.3 asset pipeline end-to-end: builder packs Crytek Sponza scene from .glb into NEOPAK, runtime loads packs, creates entities/materials from scene manifest, renders textured 3D scene with Blinn-Phong shading, normal mapping, interactive FPS camera, and touch zones. Progressive loading via pack stacking (base + full quality). Visible on promo site.

Phase does NOT include: PBR shading, entity hierarchy, deferred rendering, shadow mapping, virtual joystick UI, audio, frustum culling, LOD.

</domain>

<decisions>
## Implementation Decisions

### Scene & model
- Crytek Sponza (glTF) from KhronosGroup/glTF-Sample-Assets — ~260K triangles, ~25 materials, ~70 meshes
- All meshes, all textures, full quality — no downscaling
- Only Sponza, no additional objects in the scene
- Git LFS for source assets (assets/sponza/) — CI with LFS caching to stay within free tier

### Builder: nt_glb_scene API (parse → inspect → selective extract)
- New `nt_glb_scene_t` struct (NOT `nt_scene_t` — reserved for future game scene format)
- `nt_builder_parse_glb_scene(&scene, "sponza.glb")` — parses full .glb, exposes nodes/meshes/materials/textures
- Developer iterates meshes and decides: which to include, resource_id naming, stream layout
- Each primitive exported as separate mesh entry (handles multi-primitive meshes)
- Textures NOT auto-extracted — developer adds them separately through existing `nt_builder_add_texture()` with own format/compression settings
- `nt_glb_material_t` exposes texture references (diffuse, normal, specular indices) so developer knows which textures each material needs
- `nt_builder_add_scene_mesh(pack, &scene, mesh_index, resource_id, &mesh_opts)` — selective mesh extraction
- `nt_builder_free_glb_scene(&scene)` cleanup

### Builder: tangent support
- Tangent mode configurable per mesh via `nt_mesh_opts_t`:
  - `NT_TANGENT_AUTO` (default) — extract from glTF if present, compute MikkTSpace if not
  - `NT_TANGENT_COMPUTE` — always compute (ignore glTF tangents)
  - `NT_TANGENT_REQUIRE` — error if glTF doesn't have tangents
  - `NT_TANGENT_NONE` — skip tangent attribute (for meshes without normal mapping)
- TANGENT stored as vec4 (xyz = direction, w = bitangent sign) — standard glTF convention
- Adds ~4MB to Sponza vertex data (+16 bytes per vertex)

### NT_ASSET_BLOB (new asset type)
- Generic binary data asset — builder packs raw bytes, game interprets
- Minimal header: magic + version (pack entry already has size)
- No GPU activator — game gets raw data pointer via `nt_resource_get_blob(handle)`
- Used for scene manifest and any future game data (configs, tables, etc.)

### Scene manifest
- Binary blob asset in pack containing per-node data: resource_id (mesh), resource_id (texture per slot), transform (mat4), material params
- Game reads manifest at load time, creates entities in loop
- Manifest generated by builder from `nt_glb_scene.nodes[]`

### Two packs with progressive loading
- `sponza_base.ntpack` (priority 10): compressed assets — float16 positions, int8 normals, smaller textures. ~2-5MB, loads fast
- `sponza_full.ntpack` (priority 20): full quality — float32 positions, full-res textures. ~30-50MB, loads in background
- Same resource_ids in both packs — pack stacking auto-swaps when full pack ready
- Demonstrates pack stacking feature (Phase 24) in real use

### Shader features: Blinn-Phong + normal mapping
- One directional light + ambient — parameters in Light UBO (slot 1)
- Shader permutations (NOT fallback textures) to showcase multiple shader/material combos:
  - `sponza_full.vert/frag` — diffuse + normal + specular (columns, stone)
  - `sponza_diffuse.vert/frag` — diffuse only (simple materials)
  - `sponza_alpha.vert/frag` — diffuse + alpha test (leaves, curtains)
- Shaders as pack assets (existing pattern from textured_quad)
- Each material references appropriate shader pair
- Demonstrates: different materials use different shaders on same scene

### Engine: nt_gfx_register_global_block()
- Replace hardcoded "Globals" auto-bind in `nt_gfx_gl.c:488` with registered global blocks
- `nt_gfx_register_global_block("Globals", 0)` — game registers at init
- `nt_gfx_register_global_block("Lighting", 1)` — game adds light UBO
- At pipeline creation: engine iterates registered blocks, auto-binds any found in shader
- Game-controlled, no hardcoded block names in engine

### Camera: FPS with touch zones
- FPS camera: WASD movement, LMB drag = look rotation
- Scroll wheel = speed adjustment
- Space/Shift = up/down
- Fixed starting position (hardcoded good viewpoint in atrium)
- Touch zones for mobile (code only, no visual UI):
  - Left half: drag = movement (dx → strafe, dy → forward/back)
  - Right half: drag = look rotation
  - Visual joystick pad deferred until sprite/UI rendering available
- ESC = quit (native only)

### Demo organization
- New example: `examples/sponza/` — separate from textured_quad
- `examples/sponza/build_packs.c` — builder code for both packs
- `examples/sponza/main.c` — runtime demo
- `examples/sponza/site_promo/index.md` — promo site integration
- Pool sizes: `max_entities=256`, `max_textures=128`, `max_materials=64`

### Established Patterns
- Init sequence: engine → window → input → gfx → http → fs → hash → resource → activators → material → entity → components → mesh_renderer
- Frame order: window_poll → input_poll → resource_step → material_step → transform_update → build items → sort → draw
- Pack stacking: mount with priority, same resource_ids, higher priority wins — textured_quad demonstrates with pixel/hires packs
- Context loss recovery: invalidate resources, re-register virtual packs, recreate game-owned GPU resources

- `engine/CMakeLists.txt` — No new engine modules needed; register_global_block is an addition to nt_gfx

<specifics>
## Specific Ideas

- "Хочется показать большую сцену — что движок может много и быстро рисовать, что есть меши и разные материалы"
- Parse → inspect → selective extract: developer sees every mesh and texture before adding to pack, controls resource_ids and formats
- nt_glb_scene (NOT nt_scene) — user's game scene format will be separate
- Shader permutations showcase the material system: same scene, different shaders per material group
- Two packs demonstrate pack stacking in real use, not just a test (pixel/hires was proof, this is production)
- Touch zones now, visual joystick pad later when UI rendering available
- register_global_block replaces hardcoded "Globals" — game controls all UBO conventions

</specifics>

<deferred>
## Deferred Ideas

- Runtime tangent computation at mesh load time — saves pack size, costs load time. Note as future optimization
- Virtual joystick/touchpad visual overlay — needs sprite/UI rendering system
- Loading progress bar for WASM — needs UI rendering
- PBR shading (metallic-roughness workflow) — Intel Sponza, future milestone
- Pointer lock mode for FPS camera — requestPointerLock() in web platform
- Scene auto-tour camera path — cinematic walkthrough before user input
- Frustum culling — skip off-screen entities
- LOD system — distance-based mesh/texture quality
- Compressed textures (ETC2/ASTC) — ADV-01, v1.4
- Shadow mapping — v1.4+
- Audio system — not needed for visual demo

</deferred>
