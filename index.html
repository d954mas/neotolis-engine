Neotolis Engine

Minimalist game engine for Web/WASM

Tiny output · Zero magic · Code-first · C17

Philosophy

  • Data-Oriented Design

    Sparse component storages, dense iteration, and predictable memory access patterns. Performance comes from how data is laid out, not from abstraction layers.

  • Code-First Architecture

    Render loop, system order, builder rules, and orchestration logic are all defined in game code. No heavy declarative systems or configuration files.

  • No Hidden Schedulers

    No system registry, no hidden render graph, no automatic ECS magic. The game controls the order of execution, not the engine.

  • Composable Features

    The engine is a set of modules. Your game links only the features it needs. There is no monolithic pipeline to buy into.

  • Builder Does the Heavy Work

    Offline builder validates and processes assets. Runtime stays simple -- it loads, resolves, and renders. No expensive validation in the hot path.

Features

  • Code-First Architecture — Render loop, system order, and builder rules are all defined in game code. No heavy declarative systems.
  • No Hidden Magic — No hidden scheduler, no hidden render graph, no automatic ECS magic. Explicit over implicit.
  • Composable Modules — Engine is a set of features. Your game links only the modules it needs. No monolithic pipeline.
  • Tiny WASM Output — Minimal runtime with compile-time limits and zero heap allocation in hot paths. Builds measure in kilobytes.
  • Data-Oriented Design — Sparse component storages, dense iteration, typed handles. Predictable memory access patterns.
  • Builder Does the Heavy Work — Offline builder validates and processes assets. Runtime only loads, resolves, and renders.
